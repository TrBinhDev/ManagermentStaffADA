## 0. Thay đổi lên bảng `Position` đã có sẵn — thêm `isActive`

Từ module này, `Position` bắt đầu có dữ liệu gắn theo nó nhiều hơn (mức lương, giới hạn ca) nên không thể chỉ dựa vào xóa cứng + chặn FK như trước. Bổ sung field:

```prisma
model Position {
  // ...các field cũ giữ nguyên...
  isActive Boolean @default(true)
}
```

- `isActive=false`: chỉ ẩn Position khỏi các danh sách **chọn mới** (tạo nhân viên, đổi vị trí nhân viên, rehire, xếp lịch làm việc). Không ảnh hưởng nhân viên đang giữ vị trí đó, không ảnh hưởng dữ liệu lịch sử đã có.
- Hành vi giống hệt `ManagerAccount.isActive` đang dùng (khóa tài khoản = ẩn, không xóa).
- **`PATCH /positions/:id`** (endpoint đã có từ v3) nhận thêm field `isActive?: boolean` trong body — không cần endpoint riêng.
- **`DELETE /positions/:id`** giữ nguyên là xóa cứng thật (không phải set `isActive=false`) — chỉ dùng khi Position chưa từng có dữ liệu liên quan gì. Service `remove()` (đã có check đếm `PositionHistory` từ v3) bổ sung thêm check đếm `PositionSalaryRate` — còn có mức lương nào (dù đã đóng) thì chặn xóa cứng, báo dùng `isActive=false` thay thế. Không cần check `ShiftPositionCapacity` vì bảng đó `onDelete: Cascade`, tự dọn khi xóa Position.

## 1. `PositionSalaryRate` — Lịch sử mức lương theo giờ của từng vị trí

### Schema

```prisma
model PositionSalaryRate {
  id            String    @id @default(uuid())
  positionId    String
  hourlyRate    Decimal
  effectiveFrom DateTime
  effectiveTo   DateTime? // null = đang áp dụng

  position Position @relation(fields: [positionId], references: [id], onDelete: Restrict)

  @@index([positionId])
}
```

`onDelete: Restrict` — Position từng có mức lương thì không xóa cứng được nữa (dùng `isActive=false` để ẩn thay vì xóa).

### API

**`GET /positions/:id/salary-rates`**
Quyền: [OWNER|MANAGER]. Trả về lịch sử toàn bộ mức lương của 1 Position, sắp xếp `effectiveFrom desc`.

**`POST /positions/:id/salary-rates`**
Quyền: **[OWNER]** — đặt mức lương là quyết định tài chính cấp cao, khác với các thao tác vận hành hàng ngày (chấm công, xếp lịch) mà MANAGER cần làm; giống nguyên tắc OWNER-only đang áp cho quản lý tài khoản. Body: `{ hourlyRate: number }`.
Logic (transaction):

1. Đóng mức đang mở (nếu có): `effectiveTo = hôm nay`.
2. Mở mức mới: `hourlyRate`, `effectiveFrom = hôm nay`, `effectiveTo = null`.

Không có `PATCH`/`DELETE` cho 1 dòng đã tồn tại — sửa lương chỉ có cách tạo mức mới (đóng mức cũ tự động). Lý do: tránh làm sai lệch `DailyPayment` đã snapshot theo mức cũ.

## 2. `Shift` — Danh mục ca làm việc

### Schema

```prisma
model Shift {
  id        String   @id @default(uuid())
  name      String   @unique // "Ca sáng", "Ca chiều", "Ca tối"
  startTime String   // "06:00"
  endTime   String   // "14:00"
  isActive  Boolean  @default(true)
  createdAt DateTime @default(now())
  updatedAt DateTime @updatedAt

  capacities     ShiftPositionCapacity[]
  workSchedules  WorkSchedule[]
}
```

### API — CRUD đầy đủ

**`GET /shifts`** — Quyền: [OWNER|MANAGER]. Query: `?isActive=&page=&limit=`.
**`GET /shifts/:id`** — Quyền: [OWNER|MANAGER].
**`POST /shifts`** — Quyền: [OWNER|MANAGER]. Body: `{ name, startTime, endTime }`.
**`PATCH /shifts/:id`** — Quyền: [OWNER|MANAGER]. Body: `{ name?, startTime?, endTime?, isActive? }`.
**`DELETE /shifts/:id`** — Quyền: [OWNER|MANAGER]. Chỉ xóa cứng được nếu chưa từng xuất hiện trong `WorkSchedule` — đã dùng rồi thì 409, gợi ý dùng `PATCH { isActive: false }`.

## 3. `ShiftPositionCapacity` — Giới hạn số người theo từng cặp (Ca, Vị trí)

### Schema

```prisma
model ShiftPositionCapacity {
  id         String @id @default(uuid())
  shiftId    String
  positionId String
  maxStaff   Int

  shift    Shift    @relation(fields: [shiftId], references: [id], onDelete: Cascade)
  position Position @relation(fields: [positionId], references: [id], onDelete: Cascade)

  @@unique([shiftId, positionId])
}
```

Ví dụ dữ liệu: (Ca sáng, Phục vụ) → maxStaff=3; (Ca sáng, Bếp trưởng) → maxStaff=2.

> **Mặc định:** nếu 1 cặp `(shiftId, positionId)` **chưa được khai báo** trong bảng này, xếp lịch **không bị giới hạn** — validate ở `WorkSchedule` chỉ áp dụng khi tìm thấy dòng khớp, không tìm thấy thì cho xếp thoải mái, không chặn.

### API

**`GET /shifts/:id/capacities`** — Quyền: [OWNER|MANAGER]. Trả toàn bộ giới hạn theo vị trí của 1 Shift.
**`POST /shifts/:id/capacities`** — Quyền: [OWNER|MANAGER]. Body: `{ positionId, maxStaff }`. Trùng cặp `(shiftId, positionId)` → 409.
**`PATCH /shifts/:id/capacities/:capacityId`** — Quyền: [OWNER|MANAGER]. Body: `{ maxStaff }`.
**`DELETE /shifts/:id/capacities/:capacityId`** — Quyền: [OWNER|MANAGER].

## 4. `WorkSchedule` — Lịch làm việc theo ngày

### Schema

```prisma
model WorkSchedule {
  id         String   @id @default(uuid())
  employeeId String
  shiftId    String
  workDate   DateTime @db.Date // quy ước: tính theo NGÀY KẾT THÚC ca (kể cả ca qua đêm)
  createdAt  DateTime @default(now())

  employee Employee @relation(fields: [employeeId], references: [id], onDelete: Cascade)
  shift    Shift    @relation(fields: [shiftId], references: [id], onDelete: Restrict)

  @@unique([employeeId, shiftId, workDate])
}
```

> **Quy ước ca qua đêm:** nếu Shift có `endTime < startTime` (VD 22:00 → 06:00, kết thúc hôm sau), `workDate` của dòng lịch này lấy theo **ngày kết thúc ca** (ngày hôm sau), không phải ngày bắt đầu. Áp dụng nhất quán cho cả `WorkSchedule` và `Attendance`/`DailyPayment` bên dưới.

### API

**`GET /employees/:id/work-schedule`**
Quyền: [OWNER|MANAGER]. Query: `?month=&year=`. Trả lịch cả tháng của 1 nhân viên.

**`POST /employees/:id/work-schedule/bulk`**
Quyền: [OWNER|MANAGER]. Body: `{ shiftId: string, workDates: string[] }`.
Logic (transaction, xử lý từng ngày độc lập):

0. Check `Employee.status === ACTIVE` **1 lần cho cả request** (status không đổi giữa các ngày trong cùng 1 lần gọi) — nhân viên đã nghỉ việc → từ chối toàn bộ, 400 "Nhân viên đã nghỉ việc, không thể xếp lịch". Tránh xếp lịch cho người đã nghỉ, phát hiện lỗi ngay lúc xếp thay vì đợi đến lúc chấm công mới báo.
1. Với mỗi `workDate` trong `workDates`:
   - Lấy `maxStaff` của cặp `(shiftId, employee.positionId)` từ `ShiftPositionCapacity` — **không tìm thấy dòng nào thì coi là không giới hạn, luôn cho xếp, bỏ qua bước đếm bên dưới**.
   - Có khai báo `maxStaff`: đếm số nhân viên **cùng vị trí** đã xếp vào `(shiftId, workDate)` đó (join `WorkSchedule` → `Employee` → lọc `positionId` trùng).
   - Đã đạt `maxStaff` → bỏ qua ngày này (không tạo), ghi nhận vào danh sách `rejected`.
   - Chưa đầy (hoặc không giới hạn) → tạo `WorkSchedule` cho ngày đó, ghi nhận vào danh sách `created`.
2. Trả về response: `{ created: string[], rejected: string[] }` — để FE biết chính xác ngày nào xếp được, ngày nào bị từ chối vì đầy vị trí đó trong ca.

**`PATCH /employees/:id/work-schedule/:scheduleId`**
Quyền: [OWNER|MANAGER]. Body: `{ shiftId: string }` — đổi ca cho 1 ngày đã xếp.
Logic: validate lại `maxStaff` của ca mới (cùng cách tính ở trên) trước khi đổi — đầy thì 400.

**`DELETE /employees/:id/work-schedule/:scheduleId`**
Quyền: [OWNER|MANAGER]. Gỡ lịch nếu xếp nhầm.

**`GET /work-schedule`** (tab tổng hợp — không thuộc riêng 1 nhân viên)
Quyền: [OWNER|MANAGER]. Query: `?month=&year=&shiftId=`. Trả lịch làm việc **toàn bộ nhân viên** trong tháng — dùng cho tab "Lịch làm việc" ở thanh điều hướng chính (hiển thị dạng bảng: hàng = nhân viên, cột = ngày).

## 5. `Attendance` — Chấm công thật (giờ vào + giờ ra)

### Schema

```prisma
model Attendance {
  id             String    @id @default(uuid())
  employeeId     String
  shiftId        String    // ca đã chấm công, phải khớp WorkSchedule đã xếp
  workDate       DateTime  @db.Date
  checkedInAt    DateTime?
  checkedOutAt   DateTime?
  checkedInById  String    // tài khoản quản lý bấm check-in
  checkedOutById String?   // tài khoản quản lý bấm check-out — có thể khác người check-in (khác ca trực)

  employee     Employee        @relation(fields: [employeeId], references: [id], onDelete: Restrict)
  shift        Shift           @relation(fields: [shiftId], references: [id], onDelete: Restrict)
  checkedInBy  ManagerAccount  @relation("AttendanceCheckedInBy", fields: [checkedInById], references: [id], onDelete: Restrict)
  checkedOutBy ManagerAccount? @relation("AttendanceCheckedOutBy", fields: [checkedOutById], references: [id], onDelete: Restrict)
  dailyPayment DailyPayment?

  @@unique([employeeId, workDate, shiftId]) // 1 nhân viên có thể chấm nhiều ca/ngày, nhưng mỗi ca chỉ 1 lần
}
```

> `employee` đổi thành `onDelete: Restrict` (khác Cascade như bản nháp trước) — còn dữ liệu chấm công/lương thì không cho xóa cứng `Employee` nữa, giống nguyên tắc bảo vệ dữ liệu tài chính đã áp cho `Position`. `DailyPayment` phụ thuộc `attendanceId` nên chặn ở đây là đủ, không cần chặn riêng ở `DailyPayment`.

### API

**`POST /attendance/check-in`**
Quyền: [OWNER|MANAGER]. Body: `{ employeeId: string, shiftId: string, workDate: string }`.
Logic: validate `Employee.status === ACTIVE`. **Bắt buộc phải có `WorkSchedule` khớp đúng `(employeeId, shiftId, workDate)` đã xếp từ trước** — không tìm thấy → 400 "Nhân viên chưa được xếp ca này, không thể chấm công". Tạo `Attendance` (`checkedInAt = now`, `checkedInById = req.user.id` — tài khoản đang đăng nhập) — trùng `(employeeId, workDate, shiftId)` đã có → 409.

> `workDate` bắt buộc client truyền lên (không tự suy ra "hôm nay") — vì FE luôn chấm công dựa trên 1 dòng `WorkSchedule` cụ thể đang hiển thị (đã có sẵn đúng `workDate` theo quy ước ca qua đêm tính theo ngày kết thúc), tránh sai lệch nếu chấm công gần nửa đêm.

**`PATCH /attendance/:id/check-out`**
Quyền: [OWNER|MANAGER].
Logic (transaction):

1. Validate `checkedInAt` đã có, `checkedOutAt` chưa có (400 nếu đã check-out rồi).
2. Set `checkedOutAt = now`, `checkedOutById = req.user.id` (tài khoản đang đăng nhập — có thể khác `checkedInById`).
3. Tính `DailyPayment` (xem mục 6) — tạo trong cùng transaction.

**`GET /attendance`**
Quyền: [OWNER|MANAGER]. Query: `?employeeId=&from=&to=&page=&limit=`.

## 6. `DailyPayment` — Tiền công từng ngày, tính tự động

### Schema

```prisma
model DailyPayment {
  id           String   @id @default(uuid())
  attendanceId String   @unique
  employeeId   String
  positionId   String   // snapshot vị trí THỰC TẾ nhân viên giữ tại workDate (tra qua PositionHistory) — không đổi dù sau này nhân viên đổi vị trí
  workDate     DateTime @db.Date
  hoursWorked  Decimal  // số giờ, tính chính xác tới phút (không làm tròn giờ)
  hourlyRate   Decimal  // snapshot mức lương ÁP DỤNG tại workDate — không đổi dù Position sau này đổi lương
  amount       Decimal  // hoursWorked * hourlyRate, làm tròn tới 1.000đ

  attendance Attendance @relation(fields: [attendanceId], references: [id], onDelete: Cascade)
  employee   Employee   @relation(fields: [employeeId], references: [id], onDelete: Restrict)
  position   Position   @relation(fields: [positionId], references: [id], onDelete: Restrict)

  @@index([employeeId, workDate])
}
```

### Không có API tạo/sửa tay — chỉ tự sinh từ `PATCH /attendance/:id/check-out`

**Công thức tính (chạy trong transaction của check-out):**

```
1. Tìm PositionHistory của employee có startDate <= workDate <= (endDate ?? hôm nay)
   → đây là vị trí THỰC TẾ nhân viên giữ vào đúng workDate (không phải Employee.positionId hiện tại,
     vì nhân viên có thể đã đổi vị trí sau workDate nhưng trước lúc check-out)
   → snapshot positionId này vào DailyPayment.positionId

2. Lấy positionId (từ bước 1) → tìm PositionSalaryRate đang hiệu lực tại workDate
   (effectiveFrom <= workDate AND (effectiveTo IS NULL OR effectiveTo > workDate))
   → snapshot giá trị này vào DailyPayment.hourlyRate

3. hoursWorked = (checkedOutAt - checkedInAt tính bằng giây) / 3600
   — dùng thư viện Decimal-safe (VD decimal.js), KHÔNG dùng phép toán Float thường

4. amount = hoursWorked * hourlyRate → làm tròn tới **1.000đ gần nhất** (Math.round(amount / 1000) * 1000)
```

### API đọc dữ liệu

**`GET /employees/:id/payments`**
Quyền: [OWNER|MANAGER]. Query: `?month=&year=`.
Response: `{ data: DailyPayment[], totalAmount: number, totalHours: number }` — `totalAmount = SUM(amount)` tính real-time mỗi lần gọi, không cần job "chốt lương" cuối tháng.

**`GET /payments`** (tab tổng hợp — không thuộc riêng 1 nhân viên)
Quyền: [OWNER|MANAGER]. Query: `?month=&year=&employeeId=`. Trả tổng lương **toàn bộ nhân viên** trong tháng — dùng cho tab "Lương" ở thanh điều hướng chính. Response: `{ data: [{employeeId, fullName, totalAmount, totalHours}], grandTotal: number }`.

## Tổng kết các quyết định nghiệp vụ đã chốt cho module này

1. Lương gắn theo `Position`, không phải theo từng `Employee` riêng lẻ — đổi lương là đổi cho cả vị trí.
2. Lương có lịch sử (`PositionSalaryRate`), không sửa đè — sửa nhầm thì tạo mức mới, không xóa mức cũ.
3. Giới hạn nhân sự theo ca tính theo **từng cặp (Ca, Vị trí)**; cặp nào chưa khai báo `maxStaff` thì mặc định KHÔNG giới hạn.
4. Lương tính theo **giờ chấm công thực tế** (check-in/check-out), không theo lịch đã xếp.
5. Lương tính **ngay khi check-out mỗi ngày**, không đợi chốt cuối tháng — xem tổng tháng chỉ là `SUM` real-time.
6. `hourlyRate` snapshot vào từng `DailyPayment` — đổi lương sau này không ảnh hưởng ngày công cũ đã tính.
7. `Position`/`Shift` không xóa cứng được nếu đã có dữ liệu liên quan (lương, lịch, nhân viên) — dùng `isActive=false` để ẩn thay vì xóa. `isActive=false` chỉ chặn chọn mới, không ảnh hưởng ai đang giữ/đã xếp.
8. Xếp lịch làm việc hỗ trợ chọn nhiều ngày cùng lúc (`bulk`), validate riêng từng ngày, trả rõ ngày nào thành công/bị từ chối.
9. **Check-in bắt buộc khớp với `WorkSchedule` đã xếp trước** — không có lịch thì không chấm công được, tránh chấm công tùy tiện.
10. **Ca qua đêm tính `workDate` theo ngày kết thúc ca** — áp dụng nhất quán cho `WorkSchedule`, `Attendance`, `DailyPayment`.
11. **`amount` làm tròn tới 1.000đ gần nhất** (không giữ số lẻ dưới nghìn).
12. Lịch làm việc và lương đều có **2 nơi xem**: tab tổng hợp toàn nhà hàng (`GET /work-schedule`, `GET /payments`) và tab con trong trang chi tiết từng nhân viên (`GET /employees/:id/work-schedule`, `GET /employees/:id/payments`).
13. `Position` (bảng cũ v3) bổ sung `isActive` — ẩn khỏi lựa chọn mới, không ảnh hưởng dữ liệu/nhân viên đang giữ. Xóa cứng `Position` vẫn bị chặn nếu còn `PositionHistory` hoặc `PositionSalaryRate`.
14. Lương tính theo **vị trí thực tế nhân viên giữ tại `workDate`** (tra qua `PositionHistory`), không phải `Employee.positionId` hiện tại — vì vị trí có thể đổi sau `workDate` nhưng trước lúc check-out. `positionId` này cũng được snapshot vào `DailyPayment` (giống `hourlyRate`).
15. `Attendance` tách riêng `checkedInById` và `checkedOutById` — người bấm check-in và check-out có thể là 2 tài khoản quản lý khác nhau (khác ca trực).
16. `Employee` (bảng cũ) không cho xóa cứng nữa nếu đã có dữ liệu chấm công/lương (`Attendance.employee` đổi thành `onDelete: Restrict`, khác Cascade như các bảng lịch sử khác) — bảo vệ dữ liệu tài chính đã phát sinh thật.
17. Riêng **tạo mức lương mới** (`POST /positions/:id/salary-rates`) giới hạn **OWNER-only** — khác các API còn lại của module này vẫn `[OWNER|MANAGER]`, vì đây là quyết định tài chính cấp cao chứ không phải vận hành hàng ngày.
18. `POST /employees/:id/work-schedule/bulk` chặn ngay từ đầu nếu `Employee.status !== ACTIVE` — không đợi đến lúc chấm công mới phát hiện nhân viên đã nghỉ việc.
